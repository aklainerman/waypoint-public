# Waypoint — DoW CRM + Budget App

A frozen snapshot from a working production app. Not maintained.

Waypoint is an opinionated DoD-budget exploration + outreach CRM. It
ships preloaded with every FY27 Program Element, O&M Service Activity
Group, and Procurement Line that appears in the publicly released
Defense budget justification books, plus a Hill Ops module that
tracks every member of the 119th Congress and their committees.
Drop your own contacts/solicitations/letters on top and you have a
working DoD-outreach CRM in an afternoon.

## What's in it

- **2,143 DoD FY27 RDT&E Program Elements** with budget figures (PB
  request, PY enacted, FYDP), prior-year actuals, owning organization,
  and unclassified narratives extracted from the J-Books.
- **389 O&M SAGs** (Service Activity Groups) with FY27 mandatory amounts
  and Attachment 1 provenance.
- **~2,500 Procurement Lines** with FY27 line-item budgets and
  Exhibit P-40 narratives.
- **PE ↔ Office mapping** — 1,798 entities pre-tagged with their
  probable DoD subordinate-organization owner, surfaced as the
  Budget tab's primary organizing axis. **The mapping is a starting
  point, not gospel**: it was generated by an LLM pass against the
  public J-Books and is roughly 80% accurate at the entity level. The
  structure of the org tree (which organizations exist, parent →
  child relationships) is solid; only the PE-to-org assignments are
  approximate. Fix individual tags via the Edit Office modal as you
  spot wrong assignments — the data model expects this kind of
  iterative refinement.
- **Hill Ops** — every U.S. Senator + Representative in the 119th
  Congress, their committee assignments, chair status, party, state,
  and the committees themselves with chair/ranking info. Free of CRM
  data on first deploy; you add meetings and requests as you go.
- **A complete CRM** for Offices, Contacts, Solicitations (proposals),
  Letters (support letters), and Washops (Hill meetings). Empty on
  first deploy; you fill with your own data.
- **Optional: Scout** — an LLM-driven contact research agent that uses
  Anthropic + free public sources (SAM.gov, USAspending.gov, DVIDS,
  general web) plus optional Apollo enrichment to find DoD program
  managers and contracting officers. Requires `ANTHROPIC_API_KEY`;
  works without it (Scout tab hides itself entirely).

## Setup

You have two ways to install Waypoint. Both end at the same working app.

### Option 1: Claude-driven setup (recommended, ~30 min)

If you have access to Claude with the Supabase and Netlify MCPs
connected, Claude can drive almost the entire install for you. You
create the accounts and answer a few prompts; Claude handles the
SQL, deploy, and wiring.

See **[docs/CLAUDE_SETUP.md](docs/CLAUDE_SETUP.md)** for the full
walkthrough.

### Option 2: Manual install (~3 hours)

Drive every step yourself. Requires `psql` on your machine for the
larger seed files (Supabase Studio's SQL editor can't handle them).

See the **Manual install** section below.

## What you'll do yourself either way

Regardless of which path you take, these steps stay manual:

1. **Create a Supabase account + project.** Free tier is fine.
2. **Create a Netlify account.** Free tier is fine.
3. **Fork this repo to your GitHub.** (Or use the Claude path with a
   GitHub MCP connected and Claude can fork for you.)
4. **Click the magic-link email** to sign in once everything is deployed.
5. **(Optional) Get API keys** for Scout (Anthropic), Apollo, SAM.gov,
   DVIDS, Congress.gov — see the API keys table below.

Everything else is automatable.

## Manual install

For Option 2 (manual). Skip to [API keys](#api-keys) if you're using
Option 1.

### 1. Create the Supabase project

[supabase.com/dashboard](https://supabase.com/dashboard) → New Project.
Pick a name, region, and a strong database password. Save the
password — you need it for psql. From Settings → API note the
project URL, anon key, and service-role key.

### 2. Install psql

The schema dump and several seed files exceed what Supabase Studio's
SQL editor can paste in one go (~1 MB limit). You need the PostgreSQL
17 command-line client. Pick the installer that matches your OS:

**Windows.** Install from
[enterprisedb.com/downloads/postgres](https://www.enterprisedb.com/downloads/postgres).
You only need the "Command Line Tools" component during install; uncheck
the rest. Add `C:\Program Files\PostgreSQL\17\bin\` to your PATH.

**macOS.** Either:
- Homebrew: `brew install libpq && brew link --force libpq` (libpq is the
  client-only package, ~30 MB, no server). After linking, `psql` is on
  your PATH.
- Or [Postgres.app](https://postgresapp.com): full server bundle with
  GUI; you only need the bundled `psql` binary. Add
  `/Applications/Postgres.app/Contents/Versions/latest/bin` to PATH.

**Linux.** Use your distro's package manager:
- Debian / Ubuntu: `sudo apt install postgresql-client-17`
- Fedora / RHEL: `sudo dnf install postgresql17`
- Arch: `sudo pacman -S postgresql-libs`

Verify on any OS: `psql --version` should print `psql (PostgreSQL) 17.x`.

### 3. Apply schema migrations

Apply these four files in order via psql. Substitute your project ref
and database password. The commands below use forward-slash paths,
which work on every shell (PowerShell, bash, zsh, fish).

**Windows (PowerShell).**

```powershell
$env:PGPASSWORD = "<your db password>"
$DB_URL = "postgresql://postgres@db.<your-project-ref>.supabase.co:5432/postgres"

psql $DB_URL -v ON_ERROR_STOP=1 -f supabase/migrations/000_initial_schema.sql
psql $DB_URL -v ON_ERROR_STOP=1 -f supabase/migrations/001_rls.sql
psql $DB_URL -v ON_ERROR_STOP=1 -f supabase/migrations/002_is_email_allowed.sql
psql $DB_URL -v ON_ERROR_STOP=1 -f supabase/migrations/003_auth_trigger.sql
```

**macOS / Linux (bash, zsh).**

```bash
export PGPASSWORD="<your db password>"
DB_URL="postgresql://postgres@db.<your-project-ref>.supabase.co:5432/postgres"

psql "$DB_URL" -v ON_ERROR_STOP=1 -f supabase/migrations/000_initial_schema.sql
psql "$DB_URL" -v ON_ERROR_STOP=1 -f supabase/migrations/001_rls.sql
psql "$DB_URL" -v ON_ERROR_STOP=1 -f supabase/migrations/002_is_email_allowed.sql
psql "$DB_URL" -v ON_ERROR_STOP=1 -f supabase/migrations/003_auth_trigger.sql
```

The fourth migration (`003_auth_trigger.sql`) wires the
`handle_new_user` trigger onto `auth.users`. Without it, the first
user to sign in lands as a viewer and the first-user-becomes-admin
logic never fires.

If your Supabase project requires the pooler (most do for IPv4-only
machines), swap the host to
`aws-1-<your-region>.pooler.supabase.com:5432` and the username to
`postgres.<your-project-ref>`. New projects created in 2025+ use
`aws-1-*`; older projects use `aws-0-*`. If you get
`Tenant or user not found`, flip the digit.

For a multi-environment deploy (prod + stage + demo on three Supabase
projects), see [docs/DEPLOY.md](docs/DEPLOY.md) for the optional RLS
files in `supabase/migrations/optional/`.

### 4. Apply budget seed data

The seed lives in `supabase/seed/budget/` as 110+ SQL files
(large tables are pre-chunked). Apply them in lexicographic order.

**Windows (PowerShell).**

```powershell
Get-ChildItem supabase/seed/budget/*.sql | Sort-Object Name | ForEach-Object {
    Write-Host "Applying $($_.Name)..."
    psql $DB_URL -v ON_ERROR_STOP=1 -f $_.FullName
    if ($LASTEXITCODE -ne 0) { Write-Host "FAILED on $($_.Name)"; break }
}
```

**macOS / Linux (bash, zsh).**

```bash
for f in $(ls -1 supabase/seed/budget/*.sql | sort); do
    echo "Applying $(basename "$f")..."
    psql "$DB_URL" -v ON_ERROR_STOP=1 -f "$f" || { echo "FAILED on $f"; break; }
done
```

Total runtime: 5–10 minutes via psql over a pooler connection; ~30
seconds via direct Postgres if your network supports it.

**Faster alternative — psycopg2 (works identically on every OS).**

If you have Python with `psycopg2-binary` installed
(`python3 -m pip install --user psycopg2-binary`), the apply runs ~10×
faster because it skips per-file process spawn:

```bash
python3 scripts/apply_seeds.py
```

…where `scripts/apply_seeds.py` reads `DB_URL` from an env var. (Not
shipped in the snapshot; use the inline script in
`.claude/skills/setup-waypoint/SKILL.md` Phase 4 as a template — it's
the same code Claude uses in the Claude-driven path.)

See [supabase/seed/budget/README.md](supabase/seed/budget/README.md)
for per-file row counts.

### 5. Deploy to Netlify

Push your fork to GitHub if you haven't. Then:

1. [app.netlify.com](https://app.netlify.com) → Add new site → Import
   an existing project.
2. Connect to GitHub, pick your fork.
3. Leave build settings default — `netlify.toml` specifies them.
4. Before clicking Deploy, click "Show advanced" → "New variable" and
   add the env vars listed in [API keys](#api-keys). Minimum:
   `WAYPOINT_ENV=prod`, `SUPABASE_URL`, `SUPABASE_ANON_KEY`,
   `SUPABASE_SERVICE_ROLE_KEY`.

   **Important:** do NOT mark `SUPABASE_URL` or `SUPABASE_ANON_KEY` as
   secret. Netlify drops secret-marked env vars from the function build
   environment, and `functions/config.js` reads both at build time. Only
   `SUPABASE_SERVICE_ROLE_KEY` and the Anthropic/Apollo keys belong as
   secrets.

5. Click Deploy. Build takes ~30 seconds.

### 5b. Configure Supabase Auth Site URL

After the Netlify deploy returns its URL (e.g.
`https://your-site.netlify.app`), update Supabase so magic-link
emails embed that URL as the redirect target.

1. Open your Supabase project dashboard →
   **Authentication** → **URL Configuration**.
2. **Site URL**: set to your Netlify URL (no trailing slash).
3. **Redirect URLs**: add two entries, one per line:
   ```
   https://your-site.netlify.app
   https://your-site.netlify.app/**
   ```
4. Click **Save**.

Why this matters: Supabase's default Site URL is
`http://localhost:3000`. Magic-link emails embed Site URL as the
redirect target, so without this step, clicking the email link
bounces you to localhost. The change is read live on each magic-link
send; no Netlify redeploy needed.

If you intend to send more than ~4 magic-link emails per hour, also
configure custom SMTP under **Authentication → Emails → SMTP Settings**.
The built-in Supabase SMTP is rate-limited and unbranded
(`noreply@mail.supabase.io`); fine for solo testing, not for higher
volume.

### 6. Add yourself to the allowlist

In Supabase SQL Editor:

```sql
INSERT INTO auth_allowlist (email) VALUES ('you@example.com');
```

Without this, the magic-link sign-in flow rejects your email pre-send.

### 7. Sign in

Open your Netlify site URL, enter your allowlisted email, click Send
magic link. Click the link in the email. You land signed-in on the
Dashboard. The Budget tab and Hill Ops tab populate from the seed;
CRM tabs start empty.

Done.

## API keys

| Key | Required? | Purpose | Where to get it |
|---|---|---|---|
| `WAYPOINT_ENV` | Yes | One of `prod` / `stage` / `demo`. Gates Demo mode. | Set per Netlify site. |
| `SUPABASE_URL` *or* `SUPABASE_DATABASE_URL` | Yes (either) | Backend connection. | Your Supabase project, Settings → API. |
| `SUPABASE_ANON_KEY` *or* `SUPABASE_PUBLISHABLE_KEY` | Yes (either) | Backend anon JWT. | Same. |
| `SUPABASE_SERVICE_ROLE_KEY` | Yes (functions only) | RLS-bypass for daemon writes. **Never** expose to the browser. | Same. |
| `ANTHROPIC_API_KEY` | Optional | Powers Scout's agent loop. Default model `claude-sonnet-4-5`. | [console.anthropic.com](https://console.anthropic.com). Pay-per-token. |
| `APOLLO_API_KEY` | Optional | Scout email + phone enrichment. ~1 credit per matched contact. | [apollo.io](https://apollo.io). Paid plan required. |
| `SAM_GOV_API_KEY` | Optional | Federal contract confirmation. Hard daily rate limit. | [sam.gov/data-services](https://sam.gov/data-services). Free. |
| `DVIDS_API_KEY` | Optional | Military news + imagery context. Generous quota. | [api.dvidshub.net](https://api.dvidshub.net). Free. |
| `CONGRESS_GOV_API_KEY` | Optional | Hill members nightly sync (separate from Scout). | [api.congress.gov](https://api.congress.gov). Free. |

See [`.claude/skills/setup-waypoint/references/env_vars.md`](.claude/skills/setup-waypoint/references/env_vars.md)
for the complete enumeration including model overrides, Apollo webhook
configuration, and site-password fallback.

**The Netlify Supabase integration's env-var names vary by integration
version.** Older versions set `SUPABASE_URL` + `SUPABASE_ANON_KEY`;
newer versions set `SUPABASE_DATABASE_URL` + `SUPABASE_PUBLISHABLE_KEY`.
`functions/config.js` accepts either pair — set whichever yours
provides; you don't need to set both.

Without `ANTHROPIC_API_KEY`, the Scout tab is hidden and any direct
hit to `/.netlify/functions/scout*` returns `503 {"error":
"scout_disabled"}`. The rest of the app works fine. See
[`docs/SCOUT.md`](docs/SCOUT.md) for the full explanation of how the
graceful degradation works (config flag → client UI gate → function
503 — three layers of defense).

## Setting up PDF deep-links (optional)

The Budget tab has "Source" buttons that deep-link to specific pages of
DoD budget justification books (J-Books). By default these 404 because
the app expects PDFs hosted in your Supabase Storage bucket named
`budget-jbooks`. To enable:

1. Open [`supabase/seed/budget/PDF_MANIFEST.md`](supabase/seed/budget/PDF_MANIFEST.md) — it lists every PDF
   the seed data references with its exact expected filename and the
   `comptroller.defense.gov` source URL.
2. Download each PDF from
   [comptroller.defense.gov/Budget-Materials/](https://comptroller.defense.gov/Budget-Materials/).
3. Create a Supabase Storage bucket named `budget-jbooks` with
   public-read access.
4. Upload each PDF preserving the exact filename from the manifest.

The rest of the app works without these — only the deep-link buttons
404. You can defer this setup indefinitely; it's a one-time cost when
(and if) you want Source buttons to resolve.

## Running stage or a public demo (optional)

This template ships with three-environment support via the
`WAYPOINT_ENV` env var: `prod`, `stage`, and `demo`. The Quickstart
above deploys a single `prod` instance — that's the minimal viable
deployment. If you want a stage environment for testing changes
before they hit prod, or a public read-only demo of your data, see
[`docs/DEPLOY.md`](docs/DEPLOY.md) for the multi-Netlify-site
pattern. The corresponding optional RLS files live at
`supabase/migrations/optional/rls_stage.sql` and `rls_demo.sql`.

Demo mode in particular implements a four-layer write defense
(CSS hide + SDK write-blocker + RLS deny-write + function 404)
documented in [`docs/DEMO.md`](docs/DEMO.md), preserved here as a
teaching artifact even though no demo deployment ships with this
template.

## Want to use something other than Netlify / Supabase?

See the **"Swap vendors? Honest coupling notes"** section at the
bottom of [`docs/DEPLOY.md`](docs/DEPLOY.md). Short version:
- **Easy:** swap Netlify for Vercel or Cloudflare Pages (~few hours,
  no app rewrite — mostly config-file + function-handler shape changes).
- **Moderate:** self-host Supabase via the upstream Docker stack
  (~a weekend, no code changes — same SDK + REST contract).
- **Hard:** swap Supabase for Firebase / Amplify / something
  fundamentally different (~50% rewrite of `js/db/*`, full re-architect
  of the RLS-based access model and the 4-layer demo write defense).
